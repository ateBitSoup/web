1. Introduction: What are Git and GitHub?

Before diving in, let's understand the two key players: Git and GitHub.

1.1. What is Git?

Git is a **distributed version control system (DVCS)**. Think of it as a powerful "Save As" feature for your entire project, but much, much smarter. It allows you to:

  • Track changes to your files over time.
  • Revert to previous versions if something goes wrong.
  • Work on different features simultaneously without interfering with each other (using branches).
  • Collaborate with others on the same project efficiently.

Git works locally on your computer. You don't need an internet connection to use most of its features once a project is on your machine.

1.2. What is GitHub?

GitHub is a **web-based platform** that provides hosting for Git repositories. It's like a social network for developers, built around Git. GitHub adds features on top of Git, such as:

  • A web interface to view your code and its history.
  • Tools for collaboration like issue tracking, pull requests, and code reviews.
  • Project management features.
  • A place to discover and contribute to open-source projects.
  • Backup for your code in the cloud.

In short: Git is the tool, GitHub is the service/platform.

Important Clarification: GitHub and `apt update`

As mentioned at the very beginning, GitHub repositories are for source code, not for Linux software packages in the way `sudo apt update` understands them.

sudo apt update refreshes your local list of available software from APT repositories (like those provided by Ubuntu or PPAs). These APT repositories contain pre-compiled binary packages (.deb files).

A GitHub repository might contain the source code for a program that you can then compile. It might also contain scripts or instructions on how to add a PPA to your system. However, GitHub itself is not an APT repository source.

2. Setting Up Your Environment

2.1. Installing Git on Ubuntu

Most Ubuntu systems come with Git pre-installed. You can check by opening your terminal (Ctrl+Alt+T) and typing:

git --version

If it's installed, you'll see the version number. If not, or if you want the latest version, install it with:

sudo apt update
sudo apt install git

2.2. Creating a GitHub Account

If you don't have one already, go to https://github.com/ and sign up for a free account. Choose a username wisely, as it will be part of your public profile and repository URLs.

2.3. Configuring Git Locally

After installing Git, you need to tell it your name and email address. This information will be associated with your commits.

git config --global user.name "Your Name"
git config --global user.email "your.email@example.com"

Replace "Your Name" and "your.email@example.com" with your actual name and the email address you used for GitHub. The --global flag means this configuration will apply to all your Git projects on this computer.

2.4. (Recommended) Setting up SSH Keys for GitHub

When you interact with GitHub (e.g., pushing your code), you need to authenticate. You can use HTTPS (username/password or personal access token) or SSH keys. SSH keys are generally more secure and convenient once set up.

  1. Check for existing SSH keys: 
    ls -al ~/.ssh
    Look for files named id_rsa.pub, id_ecdsa.pub, or id_ed25519.pub. If you have one, you can usually use it.
  2. Generate a new SSH key if needed: 
    ssh-keygen -t ed25519 -C "your.email@example.com"
    (Using Ed25519 is modern and recommended). Press Enter to accept default file locations and optionally set a passphrase for extra security.
  3. Add your SSH key to the ssh-agent: 
    eval "$(ssh-agent -s)"
ssh-add ~/.ssh/id_ed25519
    (Replace id_ed25519 if you used a different name or type).
  4. Add the SSH key to your GitHub account:

    Copy the public key content:

    cat ~/.ssh/id_ed25519.pub
    Select and copy the entire output (starts with ssh-ed25519... and ends with your email).

    Then, on GitHub:

    1. Go to your GitHub Settings (click your profile picture in the top-right, then "Settings").
    2. In the left sidebar, click "SSH and GPG keys".
    3. Click "New SSH key" or "Add SSH key".
    4. Give it a descriptive "Title" (e.g., "Ubuntu Desktop").
    5. Paste your copied public key into the "Key" field.
    6. Click "Add SSH key".
  5. Test the connection: 
    ssh -T git@github.com
    You might see a warning about the authenticity of the host. Type "yes". If successful, you'll see a message like: "Hi username! You've successfully authenticated, but GitHub does not provide shell access."

3. Working with Repositories

A **repository** (or "repo") is essentially a project's folder that Git tracks. It contains all your project files and the entire history of changes.

3.1. Option 1: Creating a New Repository on GitHub First

  1. Log in to GitHub.
  2. Click the "+" icon in the top-right corner and select "New repository".
  3. Fill in the details:
    • Repository name: Choose a short, descriptive name (e.g., my-first-project).
    • Description (optional): A brief explanation of your project.
    • Public/Private: Public repos are visible to everyone. Private repos are only visible to you and collaborators you invite. Free accounts now get unlimited private repos.
    • Initialize this repository with:
      • Add a README file: Highly recommended. A README is the first file people see and usually contains information about the project.
      • Add .gitignore: Choose a template if you know your project type (e.g., Python, Node). A .gitignore file tells Git which files or folders to ignore (like temporary files, compiled code, or sensitive credentials).
      • Choose a license (optional): If it's an open-source project, pick a license.
  4. Click "Create repository".

Now you have a repository on GitHub! To work on it locally, you'll **clone** it.

On your new repository's page on GitHub, click the green "Code" button. Make sure "SSH" is selected (if you set up SSH keys) or "HTTPS". Copy the URL provided.

In your Ubuntu terminal, navigate to where you want to store your projects (e.g., cd ~/Projects) and run:

git clone git@github.com:YourUsername/my-first-project.git

(Replace YourUsername/my-first-project.git with the URL you copied).

This creates a folder named my-first-project on your computer, containing all the files from the GitHub repository, and it's already linked to the remote GitHub repository.

3.2. Option 2: Creating a New Repository Locally First

If you already have a project folder on your computer that you want to turn into a Git repository and then push to GitHub:

  1. Navigate to your project folder in the terminal: 
    cd /path/to/your/existing-project
  2. Initialize a new Git repository: 
    git init
    This creates a hidden .git subfolder in your project directory. This is where Git stores all its tracking information.
  3. (Optional but recommended) Create a .gitignore file:

    Create a file named .gitignore in your project's root directory. Add patterns for files/folders you don't want to track. For example:

    # .gitignore example
*.log
tmp/
node_modules/
__pycache__/
.DS_Store
secret_keys.txt
  4. Add your project files to Git's tracking (staging): 
    git add .
    (The dot . means "add all files and folders in the current directory and subdirectories").
  5. Make your first commit: 
    git commit -m "Initial commit"
  6. Create a new repository on GitHub: Follow steps 1 & 2 from "Option 1" above, BUT **DO NOT** initialize it with a README, .gitignore, or license. You want an empty repository on GitHub.
  7. Link your local repository to the remote GitHub repository:

    GitHub will provide you with commands for an existing local repository. They will look something like this (copy the SSH or HTTPS URL for YOUR new empty repo):

    git remote add origin git@github.com:YourUsername/your-new-repo-name.git
git branch -M main  # Or master, depending on GitHub's default
git push -u origin main
    • git remote add origin <URL>: Tells your local Git repo where the "origin" (the remote version on GitHub) is.
    • git branch -M main: Renames your default local branch to main (GitHub's current default). Historically, it was master.
    • git push -u origin main: Pushes your local main branch to the origin remote and sets it up to track.

4. The Basic Git Workflow

Once your repository is set up (either cloned or pushed initially), the typical daily workflow involves these steps:

4.1. Check Status: `git status`

This is your go-to command. It shows you the state of your working directory and staging area: which files are modified, which are new and untracked, which are staged for commit, etc.

git status

4.2. Get Latest Changes: `git pull`

If you're collaborating or working on multiple computers, always start by pulling the latest changes from the remote repository to avoid conflicts:

git pull origin main

(Replace main with your primary branch name if different, e.g., master). This fetches changes from origin (GitHub) and merges them into your current local branch.

4.3. Make Your Changes

Edit your files, add new files, delete files, etc., using your favorite code editor or IDE.

4.4. Stage Changes: `git add`

After making changes, you need to tell Git which changes you want to include in your next "snapshot" (commit). This is called "staging."

  • To stage a specific file: git add path/to/your/file.txt
  • To stage all modified and new files in the current directory and subdirectories: git add .
  • To stage all tracked files that have been modified or deleted: git add -u

Run git status again to see what's staged.

4.5. Commit Changes: `git commit`

A commit is a snapshot of your staged changes. Each commit has a unique ID and a message describing the changes.

git commit -m "A clear, concise message describing your changes"

Good commit messages are important! They form the history of your project.

If you omit -m "message", Git will open your default text editor for you to write a longer commit message.

4.6. Push Changes to GitHub: `git push`

After committing your changes locally, you need to send them to the remote repository on GitHub:

git push origin main

(Again, replace main if your branch is named differently).

If you set up SSH, this should work without prompting for credentials. If using HTTPS, you might be asked for your GitHub username and a Personal Access Token (PAT) – passwords for Git operations over HTTPS are deprecated.

Summary of the cycle:
  1. git pull (to get updates)
  2. Edit files
  3. git add <files> (to stage)
  4. git commit -m "message" (to save snapshot locally)
  5. git push (to send to GitHub)
Repeat as you work. Use git status frequently!

5. Branching and Merging (The Basics)

Branches allow you to work on different features or fixes in isolation without affecting the main codebase (often the main or master branch). This is a cornerstone of Git.

5.1. What is a Branch?

A branch is essentially a movable pointer to one of your commits. When you create a branch, you're creating a new path for development.

5.2. Basic Branching Workflow

  1. List existing branches: 
    git branch
    The current branch will be marked with an asterisk (*).
  2. Create a new branch: 
    git branch new-feature-branch
    This creates the branch but doesn't switch to it.
  3. Switch to the new branch: 
    git checkout new-feature-branch
    Alternatively, create and switch in one command: 
    git checkout -b new-feature-branch
  4. Work on the branch: Make your changes, git add, and git commit as usual. These commits are now on new-feature-branch and not on main.
  5. Push the new branch to GitHub (first time only): 
    git push -u origin new-feature-branch
    The -u (or --set-upstream) links your local branch with the remote branch. Subsequent pushes from this branch can just be git push.
  6. Merge the branch (once work is complete and tested):

    First, switch back to the branch you want to merge INTO (usually main):

    git checkout main

    Make sure your main branch is up-to-date:

    git pull origin main

    Then, merge your feature branch into main:

    git merge new-feature-branch
    Git will try to automatically combine the changes. If there are conflicting changes (e.g., the same lines of code were changed differently in both branches), Git will pause and ask you to resolve the conflicts.
  7. Push the merged main branch: 
    git push origin main
  8. (Optional) Delete the feature branch (locally and remotely):

    Locally:

    git branch -d new-feature-branch
    (Use -D if it hasn't been fully merged and you're sure you want to delete it).

    Remotely:

    git push origin --delete new-feature-branch

On GitHub, feature branches are often merged via **Pull Requests**, which allow for code review before merging. This is covered briefly in section 7.

6. Cloning and Compiling from GitHub (Ubuntu Terminal)

Many open-source projects are hosted on GitHub. You can clone their source code and compile it on your Ubuntu system if build instructions are provided.

Compiling software is highly project-specific. There's no single command that works for all projects. You MUST look for instructions in the project's README.md file, INSTALL.md, CONTRIBUTING.md, or on their website/wiki.

6.1. General Steps

  1. Find the repository on GitHub.
  2. Clone the repository:

    Click the "Code" button, copy the SSH or HTTPS URL, and run:

    git clone <repository_url>
    Example: git clone https://github.com/someuser/someproject.git
  3. Navigate into the cloned directory: 
    cd someproject
  4. Look for build instructions:

    This is the most crucial step. Open and read files like:

    • README.md (often has "Build" or "Installation" sections)
    • INSTALL or INSTALL.md
    • BUILDING.md or HACKING.md

    These files will tell you:

    • Dependencies: Software packages you need to install first (e.g., build-essential, cmake, specific libraries like libsdl2-dev, python3-dev). You'll typically install these using `sudo apt install ...`.
    • Build System: What tools are used to compile (e.g., make, cmake, scons, npm, cargo, a specific script like ./configure or ./build.sh).
    • Build Commands: The exact commands to run.
  5. Install dependencies. For example, if it needs gcc, g++, and make: 
    sudo apt update
sudo apt install build-essential
    (build-essential is a meta-package that installs gcc, g++, make, and other common development tools).
  6. Follow the build commands.

6.2. Example: Compiling a Simple C Program with a Makefile

Let's imagine a GitHub repository for a simple "hello world" C program that uses a Makefile.

Project Structure (on GitHub, and after cloning):

simple-c-project/
├── Makefile
├── hello.c
└── README.md

hello.c contents:

// hello.c
#include <stdio.h>

int main() {
    printf("Hello from a compiled GitHub project!\n");
    return 0;
}

Makefile contents:

# Makefile
CC=gcc
CFLAGS=-Wall -Wextra -std=c11

TARGET=hello
SOURCES=hello.c

all: $(TARGET)

$(TARGET): $(SOURCES)
	$(CC) $(CFLAGS) -o $(TARGET) $(SOURCES)

clean:
	rm -f $(TARGET)

(This Makefile says: to build "all", build the "hello" target. To build "hello", compile "hello.c" using gcc with specified flags, and name the output "hello". The "clean" target removes the compiled program.)

Steps to clone and compile on your Ubuntu desktop:

  1. Clone (assuming this project exists at github.com/youruser/simple-c-project): 
    git clone https://github.com/youruser/simple-c-project.git
  2. Navigate into the directory: 
    cd simple-c-project
  3. Install dependencies (for C, you need a C compiler and `make`): 
    sudo apt update
sudo apt install build-essential # Installs gcc, g++, make, etc.
  4. Compile using the Makefile:

    The README.md (or common C project knowledge) would tell you to use make.

    make
    This will execute the commands in the Makefile, creating an executable file named hello.
  5. Run the compiled program: 
    ./hello
    Output: Hello from a compiled GitHub project!
  6. (Optional) Clean up compiled files: 
    make clean
    This would run the clean target in the Makefile, removing the hello executable.
Remember: This C example is specific. Python projects might use pip install -r requirements.txt and then run scripts. Node.js projects use npm install and npm start or npm run build. CMake projects use cmake . and then make. Always refer to the project's documentation!

7. Other Basic GitHub/Git Tasks

7.1. Viewing History: `git log`

Shows the commit history for the current branch. Press q to quit the log view.

git log

Useful options:

  • git log --oneline: Shows a compact, one-line summary.
  • git log --graph: Shows a text-based graph of branches and merges.
  • git log --pretty=format:"%h - %an, %ar : %s": Custom format.
  • git log -p -2: Show changes (patches) for the last 2 commits.

7.2. Ignoring Files: `.gitignore`

As mentioned earlier, a .gitignore file tells Git which files or directories to ignore (never track or show in git status as untracked).

Create this file in the root of your repository. Each line is a pattern. Examples:

# Comments start with a hash
*.log          # Ignore all files ending with .log
build/         # Ignore the entire 'build' directory
node_modules/  # Common for Node.js projects
*.tmp          # Ignore files ending with .tmp
!important.log # Exception: DO track important.log even if *.log is ignored
secrets.json   # Ignore sensitive files

GitHub provides good .gitignore templates for various languages/frameworks when you create a new repository.

Important: If a file is already tracked by Git, adding it to .gitignore won't make Git forget it. You'll need to untrack it first: git rm --cached <file>, then commit this removal, and then the .gitignore rule will apply.

7.3. Forks and Pull Requests (Brief Overview)

These are key to collaborating on GitHub, especially on projects you don't own.

  • Forking: A fork is a personal copy of someone else's repository. It lives in your GitHub account. You can freely experiment with changes in your fork without affecting the original project.
  • Pull Request (PR): If you've made changes in your fork that you think would benefit the original project, you can create a Pull Request. This is a formal proposal to the original project's maintainers to merge your changes into their repository.

    PRs allow for discussion, code review, and automated checks before changes are merged. This is the primary way to contribute to open-source projects.

The typical workflow for contributing to an open-source project:

  1. Fork the original repository on GitHub.
  2. Clone YOUR fork to your local machine.
  3. Create a new branch for your changes.
  4. Make your changes, commit them, and push the branch to YOUR fork on GitHub.
  5. Go to your fork on GitHub and open a Pull Request to the original repository.
  6. Discuss and iterate on the changes if the maintainers have feedback.
  7. Once approved, the maintainers will merge your PR.

7.4. The Importance of README.md

A README.md file (written in Markdown) is crucial. It's the front page of your repository. A good README should typically include:

  • Project title and brief description.
  • How to install or set up the project.
  • How to use or run the project.
  • How to contribute (if applicable).
  • License information.
  • Badges (build status, version, etc. - more advanced).

GitHub renders README.md nicely on your repository's main page.

8. Conclusion and Next Steps

This guide has covered the fundamentals of using Git and GitHub from your Ubuntu desktop. You've learned how to:

  • Set up Git and connect to GitHub.
  • Create and manage repositories.
  • Use the basic Git workflow (add, commit, push, pull).
  • Work with branches.
  • Clone and (generically) compile projects from GitHub.
  • Understand other key concepts like .gitignore, git log, forks, and pull requests.

What's next?

  • Practice! The more you use Git, the more comfortable you'll become. Create personal projects, even small ones, and use Git to track them.
  • Explore GitHub: Browse open-source projects. See how others structure their repositories and write READMEs.
  • Contribute: Find a small bug or documentation typo in an open-source project and try submitting a Pull Request.
  • Learn more advanced Git:
    • Rebasing (git rebase)
    • Stashing (git stash)
    • Resolving merge conflicts in detail
    • Using Git tags for releases
    • Different branching strategies (like Gitflow)
  • Graphical Git Clients: While the terminal is powerful, tools like GitKraken, Sourcetree, or VS Code's built-in Git integration can provide a visual way to interact with Git.

Happy coding and version controlling!
